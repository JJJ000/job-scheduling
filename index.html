Part 1
Orchestration

How a job is defined, triggered, scheduled, and moved through the system — from creation to action.

System Design

Triggers (API, cron, or events) feed the Scheduler, which records each run in the Job Metadata Store and pushes work to the Job Queue. The Worker Pool consumes the queue and writes outcomes to the Result Store.

Why Not Just Keep Everything in the Queue?

A queue is a great delivery mechanism — but a poor system of record. Leaning on it for durability invites whole classes of failure.

⚠️ Failure cases
  • Queues are optimized for delivery, not durability
  • Messages can be lost, duplicated, or expire during outages
  • Retries and failovers create inconsistent execution state
  • Hard to recover scheduler state after a crash
✅ Architectural principle
  • Keep an offline Job Metadata Store as the single source of truth
  • The queue is a transient delivery mechanism, not the record
  • Workers reconstruct state from durable metadata
  • Enables replay, recovery, auditing, and exactly-once coordination
Queue = transient transport  ·  Metadata Store = durable truth

Core Entities: Job vs Job Run

This separation is not just semantic — it is a fundamental architectural boundary. The Job is the static definition; a Job Run is one execution of it.

📘 Job static definition
  • Schedule / trigger
  • DAG definition
  • Retry policy
  • Owner / tenant metadata
  • Resource requirements
1 ──▶ N
🏃 Job Run one execution instance
  • Execution status
  • Retry attempts
  • Worker assignment
  • Logs / outputs
  • Timing & failure metadata

Why this separation matters

Track retries & failures independently per run
Durable audit history of every execution
Scales cleanly — Jobs are static, Runs drive the traffic
Efficient partitioning, retention & observability at scale

How the data flows

  1. 1The jobs definition already exists. When a trigger fires, INSERT a job_runs row with status=created (its idempotency key dedupes the run).
  2. 2Scheduler SELECTs runs where run_at ≤ now AND status=scheduled, then publishes a message carrying only the job_run_id.
  3. 3Broker holds the message; the run row flips to queued. The queue is transient — the job_runs store stays the source of truth.
  4. 4A worker pulls the message, rehydrates the run from the store, then leases it (UPDATE … SET status=running, worker_id=me, lease_expires_at=…) and runs the action.
  5. 5Success → ack + UPDATE status=completed. Failure → bump attempt; re-queue with backoff or route to DLQ once attempt ≥ retry_policy.max_attempts.
  6. 6Every transition writes updated_at on the run — durable audit trail, crash recovery, and queryable status, independent of the queue.
Part 2
Delivery Semantics

Once a job is running, how hard do we try to deliver it — and how do we avoid losing or duplicating work? Retries, backoff, and idempotency.

Every scheduler must choose how hard it tries to deliver a job. The three classic guarantees trade off lost work against duplicate work.

At-Most-Once
Fire & forget — may lose work, never duplicates
At-Least-Once (Retry)
Always retried — never lost, but may duplicate
Exactly-Once (Idempotency)
Retried and de-duplicated — the practical goal

Failures are normal, not exceptional — and which ones you retry is a product decision, not just an infrastructure one.

↻ Transient infrastructure failures → retry
✕ Permanent business-logic failures → do not retry
CI / CD
Retry: worker crash · VM/GPU preemption
Don't: build compile failure · flaky unit test
Web crawler
Retry: network timeout · HTTP 429 (rate limit)
Don't: invalid URL
GPU inference
Must not duplicate expensive computation

3 How to support idempotency?

The hard case: what if the first execution actually succeeded, but the acknowledgment was lost? A naive retry would run it twice.

Idempotent execution flow

  1. Worker receives a retry request
  2. Check the execution store (by idempotency key / result):
    • Already completed? → return the cached result
    • Otherwise → execute the task
  3. Persist the completion marker atomically

What a job_runs row tracks to make this work

The identity field gives every attempt a unique ID; the lease group lets a healthy worker safely steal a job from a dead one.

Part 3
Fairness Across Tenants

In a multi-tenant system, one tenant can overload the platform and starve everyone else. Fairness keeps the cluster shared.

1 What is the problem?

In a multi-tenant job-scheduling system, one tenant may submit too many jobs and consume the entire cluster — leaving everyone else waiting.

2 How to achieve fairness?

  1. Per-tenant queues
  2. Tenant quotas
  3. Weighted fair scheduling (priority queue)

Example · CI/CD

Burst vs. urgent
Tenant A submits 100,000 builds after a large monorepo merge.
Tenant B submits 10 urgent production hotfix builds.
Per-tenant queues: if Tenant B only needs 5 workers, Tenant A can borrow the unused capacity — but when B submits urgent builds, the system gives capacity back.

Example · Push notifications

Marketing vs. transactional
Tenant A launches a marketing campaign to 50M users.
Tenant B needs to send password-reset notifications.
Tenant C needs to send security alerts.
Separate queues by notification type, then apply priority + fairness: marketing can use spare capacity but must not starve transactional or security messages.

3 Handling GPU fairness

  1. Per-tenant queues
  2. Tenant quotas
  3. Weighted fair scheduling
Important — work-conserving scheduling: fairness should never waste capacity. If a fair share goes unused, let another tenant borrow it rather than letting GPUs sit idle.
Part 4
Long-Running Workflows

Durable state and resumable execution — so an hours-long job that dies at 90% doesn't have to start over.

1 Heartbeat-based fault detection

Workers can crash from machine failures, power loss, or network partitions. Without detection, a task can stay stuck in RUNNING forever.

Solution

  • Workers send periodic heartbeats while executing
  • Scheduler monitors the heartbeat timeout
  • Missing heartbeat → worker assumed dead
  • Task becomes eligible for retry on another worker
Worker ⟳ heartbeat every N s → Scheduler

Tracked by last_heartbeat_at on the job_runs row — the timestamp the scheduler watches to detect dead or stalled workers.

2 Checkpointing for long tasks

Some tasks run for hours — exporting millions of records, processing a massive video, training a model. Failing at 90% and restarting from zero wastes enormous time and compute.

Checkpoint by progress

10%→ 20%→ 30%→ 40%

Checkpoint by stage

Map→ reduce→ map→ reduce

At each ✓ checkpoint, store the intermediate result to durable storage.

How it works

  • Break the task into smaller stages or chunks
  • After each chunk, save progress to durable storage
  • On failure, restart from the latest checkpoint, not the beginning
  • A new worker loads the checkpoint and continues from the last good point

Examples

CI/CD: resume from the last deployed service / cluster shard instead of redeploying the whole fleet
Data pipeline: resume from the last processed partition, checkpoint, or Kafka offset
Sora / AI video: resume from completed scenes, frame ranges, or rendering stages

Applying Job Scheduling

The same building blocks — triggers, queues, retries, idempotency, fairness, and checkpointing — recombine to solve every use case from the top of this page.

  • Deploys checkpoint by shard — resume from the last deployed service, not the whole fleet.

  • Checkpoint by stage resumes from the last deployed shard — no full re-deploy.
  • Idempotent steps make at-least-once safe (never double-deploy a weight shard).

  • Each event enqueues a delivery job with at-least-once guarantees.
  • Failed deliveries retry with exponential backoff; exhausted ones land in a DLQ.
  • An idempotency key in the payload lets the receiver dedupe replays.
  • Per-merchant queues stop one busy account from delaying everyone's webhooks.

Delayed / run-atDurable store Exactly-onceSelective retry

  • Retry transient gateway errors; route hard declines to review, not blind retry.
